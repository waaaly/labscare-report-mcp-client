# LabsCare 报表脚本

这份 skill 是自包含的。

- 不要求外部样例目录。
- 不要求额外历史案例目录。
- 默认依赖用户当前提供的材料，加上本 skill 内置的参考文档。

如果用户当前材料和本 skill 的总结冲突，以用户当前材料为准。

## 决策优先级（强制）

处理 LabsCare 报表任务时，按下面顺序决策，后者不能推翻前者：

1. 模板文件图片里的实际占位符、内联表达式和明显版式关系
2. 预期结果图片
3. 当前任务附带的占位符说明、真实的 data结构相关的JSON、补充文字
4. 本 skill 的结构参考与历史案例

这条优先级的含义是：

- 模板里的占位符决定 JS 最终必须输出哪些 key。
- 结果图决定这些 key 的类型、循环方式和分页效果。
- 当前 data结构相关的JSON 只用来确认“这些 key 的值从 `form` / `sample` / `gauging` 哪一层取”，以及值是文本、对象还是数组。
- 当前 data结构相关的JSON 里真正的 source path 可能是当前实验室自己配置的标签名，例如 `two`、`three`，不一定和模板输出 key 同名。
- 当前任务标题、模板标题和结果图共同定义了“当前报表家族身份”；结构 JSON 里出现的近邻模板名不能自动替代这个身份。
- 历史案例只用来判断返回根、分组、续表、模板依赖级别，不用来发明这次任务的最终输出字段名。
- 当前实验室很多报表并不是追求“任意等价实现”，而是在复用某个报表家族的整套控制流骨架；一旦识别出家族，优先最小改动地跟随那个骨架，不要为了“更通用”“更稳健”“更工程化”而整段重写成另一套实现。

## 先向用户收什么

优先直接使用下面这段话术：

```text
为了尽量一次写准 LabsCare 报表 JS，请先把下面材料发给我。没有的可以直接说明“暂无”。

必需材料：
1. 模板文件图片
   - 请尽量提供全部页面。
   - 如果同一页有“说明版”和“实际模板版”，请都发给我。
2. 预期结果图片
   - 就是报告引擎最终应该生成出来的效果图。
   - 最好也提供全部页面。
3. 当前任务目标
   - 是从零写一份 JS 脚本
   - 还是修改现有脚本
   - 还是只补某一页、某个表格、某个签名区

强烈建议提供：
4. 占位符说明
   - 模板里如果有看不清的占位符，请用 Markdown、文字或截图补充说明。
5. 真实数据结构
   - 最好提供一份真实 JSON，类似 form / samples / gaugingTableList 这种数据。
   - 有这份数据我才能更稳地判断字段类型、关联对象、签名数组、曲线对象等细节。

如果你暂时没有 JSON，也可以先发：
“模板图片 + 结果图 + 占位符说明”
我可以先根据图像关系、模板逻辑和已有参考来反推脚本。
```

不要先追问用户自己也无法判断的问题，例如：

- 循环绑定名
- 模板是否一定要改
- 哪个历史案例最像

这些先由你根据材料自行判断。只有真的无法推出时，才做最小化追问。

## 核心工作流

1. 先确认用户至少给了：
   - 模板图片
   - 结果图
   - 任务目标
2. 先按模板图片逐页识别并抄出关键占位符、内联表达式和循环区字段。
   - 占位符名必须按模板原样记录。
   - 大小写、下划线、空格都不能擅自“规范化”。
3. 再结合结果图，反推：
   - 返回根是 `formJs`、单对象，还是 `outputData[]`
   - 报告级循环是什么
   - 行级循环数组是什么
   - 哪些字段必须保留原始对象/数组
4. 再读参考文档，判断当前任务属于哪种结构家族。
   - 这里只判断结构和骨架，不拿历史案例字段名直接当本次输出 key。
   - 一旦家族基本确定，优先复用该家族最近邻的已证实骨架，并尽量保持控制流、过滤位置、merge 顺序和输出组装方式一致。
5. 如果有 JSON，再用 JSON 验证：
   - 每个模板 key 来自 `form`、`sample` 还是 `gauging`
   - 每个模板 key 拿到的是文本、对象还是数组
   - 如果模板 key 在 JSON 里找不到同名字段，再检查当前实验室是否用了别的 source path 标签承载这个值
   - 这类 source path 只对当前实验室当前任务有效，不向别的案例泛化
6. 只有在这一步之后，才开始写 JS。
7. 能用简单占位符就不用模板内联 JS。
8. 只有模板必须直接访问原始数组/对象，或现有版式必须靠模板表达式控制时，才建议改模板格子。
9. 交付前必须反向核对结果图，确保表格展开、标题、签名、图片、日期都能解释。

## 参考文档怎么读

按这个顺序最稳：

1. `references/image-crosswalk.md`
   - 先判断当前任务更像“单页续表”、“每样一页”、“单对象多表”还是“模板强依赖”。
2. `references/examples.md`
   - 看每类报告的返回根、主过滤、分组方式、典型数组名。
3. `references/authoring-rules.md`
   - 写脚本时读这个。
   - 里面总结了环境事实、占位符判断、表格循环、何时用模板内联 JS、自检清单。
4. `references/patterns.md`
   - 需要具体代码骨架时读这个。
   - 例如 `partData + gaugingData + ObjAssign + getValue`。
5. `references/api-reference.md`
   - 只有需要确认 JSON 字段形状时再读。
6. `references/rule-reference.md`
   - 当你怀疑旧结论太绝对，或不同写法冲突时再读。

## 非谈判规则

- `模板图片 + JS 脚本 = 结果图`
- 在 LabsCare 报表引擎里，脚本块是否真正“产出值”，要看最后一行的表达式结果：
  - 模板层 / page 层脚本通常最后一行直接写 `outputData`、`reports`、`formJs`
  - 格子里的 `//javascript` 内联脚本通常最后一行直接写 `str`、`list.join(...)` 或别的目标表达式
  - `${...}` 这种写法本身就是在输出表达式值
  - 不要想当然依赖 `return`；很多真实样本根本不是靠 `return` 出值
- 最终输出 key 以模板里的实际占位符和内联表达式为准，不以历史案例字段名为准。
- JSON 用来确认来源层级和字段类型，不用来发明新的最终输出 key。
- 历史案例只用来判断结构和骨架，不用来扩充字段名、数组名或业务角色。
- 模板输出 key 与 JSON 里的 source path 可以不同名；不同名时必须把“输出 key”和“取值路径”分开思考。
- `gauging.two`、`gauging.three` 这类路径应视为“当前实验室配置标签”，不是跨实验室可复用规范。
- 结构 JSON 里如果没有当前任务的同名模板，只能把它当“字段形状参考”，不能自动改用 JSON 中最像的邻近模板名。
- 字段名、数组名、占位符名只能当“参考示例”，不能跨实验室硬搬。
- 没有 JSON 时不中断，先用模板图、结果图、占位符说明和内置参考继续反推。
- 模板图片里看不到的循环绑定名，如果内置参考和结果图都推不出来，再追问用户。
- 如果当前任务里已经能从 JSON、模板说明或现有脚本确定精确过滤条件，例如 `tempId`、`gaugingTemplateName`、`t_version`，就必须用精确过滤。
- 如果精确过滤条件本应存在，但当前材料里无法确认，不能静默降级成模糊匹配；要明确标成高风险，必要时追问。
- 当前任务已经明确属于某个报表家族时，精确过滤应优先锚定这个家族本身，而不是锚定结构 JSON 里最像的模板名。
- 但不要把“当前测试 JSON 里刚好出现的 ID”自动升级成正式脚本约束；像 `projectId` 是引擎运行时输入，像 `sample.tempId` 这类字段只有在当前报告家族证据明确要求时才应写入过滤。
- 如果脚本里不得不出现 `tempId === '114236...'`、`gaugingTemplateId === '...'`、流程模板 ID 等明确字面量 ID，这类写法默认视为“占位值 / 待替换值”，不是正式固定规则。
- 交付时必须明确提示用户：
  - 这个 ID 只是当前脚本占位值
  - 需要按本实验室实际模板 / 检测项 / 流程配置替换
  - 不要把它当成跨项目、跨实验室都通用的正式常量
- 如果参考答案或最近邻家族骨架只按 `gaugingTemplateName` 过滤，而没有 `sample.tempId` 约束，就不要因为测试数据里碰巧看到了 `tempId` 而额外加一道样品模板过滤。
- `gaugingTemplateName` 这类业务模板名过滤，与硬编码运行态 ID 不是一回事；前者常常是报表家族本身的一部分，后者通常不应手写进正式脚本。
- `t_changSuo`、`sampleId`、`t_version`、`gaugingTemplateName` 这类“结构键 / 分组键 / 匹配键”要比普通展示字段更严格；一旦当前家族已经明确，就优先使用家族里的精确键，不要给它们叠加宽松别名回退。
- 普通展示字段在证据足够时可以做小范围别名兼容，但结构键如果回退错了，会直接把报告分组、页头归属或明细归属做错。
- 如果模板格子写的是 `=xxx[0].val`、`Object.prototype.toString.call(...)` 或 `//javascript`，先检查脚本有没有把原始结构压坏。
- 对“单对象 + 多数据区”或“模板强依赖”报告，默认优先保留 `sample + form + gauging` 的原始合并结果，再补充显式加工字段。
- 但这条“原始合并结果优先”不是无条件默认；如果最近邻已证实骨架显示“行数组只保留当前 `gauging` + 少量补充字段”，就不要因为通用经验把整份 `form` 或 `sample` 再并进每一行。
- 只有能证明模板只吃少量白名单字段时，才把输出收缩成纯白名单壳。
- `partData()`、`gaugingData()` 这类白名单 schema 只是在定义“最终输出 key”，不等于这些 key 只能从某一层取值；真正的来源优先级仍要按当前报表家族证据决定。
- 对“每个样品一页 / `outputData[]` + `gaugingList`”这类扁平报表，优先考虑 `partData()` / `gaugingData()` / `ObjAssign()` / `getValue()` 这一类骨架，而不是从零发明另一套字段抽取语义。
- 对这类 `outputData[] + gaugingList` 报表，不要先验假设“行字段一定全部来自 gauging”；很多时候更稳的做法是先用 `form` / `sample` 铺默认值，再用当前行 `gauging` 覆盖。
- 不要把“家族骨架”“输出字段白名单”“来源优先级”“值转换规则”揉成一套新的自定义抽象一次性解决；这四件事要分开判断，否则很容易在看似更整洁的重构里把家族默认行为改掉。
- 如果已经找到最近邻家族骨架，默认目标不是“语义上差不多”，而是“在不违背当前模板/结果图的前提下，尽量小地偏离该骨架”。
- 判断一次偏差到底是“通用规则缺口”还是“当前家族例外”时，必须分开记录：
  - 通用规则缺口：会影响别的报告家族的反推方法，应写回 `SKILL.md` 或通用参考。
  - 家族例外：只在当前家族稳定复现，应写回对应家族小节，不要升级成所有报告的默认规则。
- 禁止把不同业务角色字段自动互相回退，例如“数据处理人”不能因为别的案例出现过就自动回退成“分析人”。
- 禁止把一个表格区的占位符字段回退到另一个表格区的字段，只因为它们看起来“像”。
- 禁止擅自改模板占位符拼写，例如模板是 `t_rsj` 就不能写成 `t_rsJ`。
- 除非结果图、任务说明或当前报表家族明确要求，否则不要擅自增加排序、去重或重排逻辑；很多报告默认沿用原始数据顺序。
- 如果结果图出现空白签名、空白日期、`[object Object]`、错位序号，优先怀疑“字段类型和模板写法不匹配”。

## 你最终要交付什么

默认交付物应包含：

1. LabsCare JS 脚本
2. 如果必须改模板：
   - 明确指出改哪一格
   - 给出可直接粘贴的占位符或内联 JS
3. 必要时说明：
   - 哪些字段必须保留数组/对象
   - 哪些字段是脚本预处理后直接展示
   - 哪些地方仍有待用户确认
   - 如果脚本里出现了明确字面量 ID，要单独标注这些 ID 是占位值，提醒用户替换
4. 关键占位符映射说明（可简短）
   - 每个关键占位符来自 `form`、`sample` 还是 `gauging`
   - 如果输出 key 和 source path 不同名，要同时写出两者
   - 证据来自模板、结果图还是 JSON

不要只给抽象建议。模板侧修改建议必须是可直接粘贴的代码。
并且模板层脚本、page 层脚本、格子内联 JS 都要明确写出“最后一行输出什么”。

## 什么时候停下来追问

仅在以下情况追问：

- 模板和结果图都看不出隐藏循环绑定名
- 同一个字段可能来自 `form`、`sample`、`gauging` 多处，且结果图无法反证
- 图片/签名 URL 方案在当前材料里完全无法判定

其他情况优先自行完成推断。

## 交付前自检

至少确认这几件事：

- 结果图里的每个循环区，都能对应到脚本里的某个数组字段
- 结果图里的每个签名位、图片位、日期位，都能解释来源
- 模板里的每个关键占位符，拿到的是正确类型
- 模板里的每个关键占位符，都能在 JS 输出里找到同名 key；如果没有，就必须明确说明为什么需要改模板
- 你能解释为什么返回根是 `formJs`、单对象或 `outputData[]`
- 你能解释为什么某些逻辑放脚本里，某些逻辑留在模板里
- 你能指出自己当前脚本与所选“最近邻家族骨架”相比，哪些地方保持一致，哪些地方做了必要偏离，以及每个偏离的证据是什么

## 1. 引擎与脚本外壳

这些规则来自已提炼的真实案例，适合作为默认起点：

- 常规开头会有 `load('/tools.js')`
- 常规入口会有 `var helper = get("labscareHelper")`
- 常用 API：
  - `helper.getProjectSamples(projectId)`
  - `helper.getProjectData(projectId)`
- 最终输出通常不是 `return`，而是最后一行直接写变量名或目标表达式：
  - `outputData`
  - `reports`
  - `formJs`
- 这条不只适用于模板层总脚本，也适用于 page 层脚本和格子内联 JS：
  - 模板层 / page 层脚本：最后一行写最终对象或数组
  - `//javascript` 内联脚本：最后一行写 `str`、`list.join(...)`、三元表达式结果等
  - `${...}` 内联写法：花括号里的表达式本身就是输出值
- 如果你在中间把字符串、数组、对象都算对了，但最后一行没有把目标变量/表达式写出来，模板就可能拿不到值。

### 数据清洗

真实案例里常见：

```javascript
samples = JSON.stringify(samples);
samples = JSON.parse(samples.replace(/null:/g,'"null":'))
```

也出现过更宽的写法：

```javascript
samples = JSON.parse(samples.replace(/null/g,'"null"'))
```

默认优先更保守的 `null:` 定向替换。

### `procedures` / `form` 取值风格不是唯一的

都是真实存在的：

```javascript
var procedure = procedures.get(processId);
var form = procedure.get('processes').get(templateld).get('form');
```

```javascript
var procedure = procedures[processId];
var form = procedure.processes[templateld].form;
```

```javascript
var form = getForm(procedures);
```

规则：

- 同一个脚本内部尽量保持一种风格。
- 优先跟随当前任务最接近的结构参考，不要强行统一。

### `templateld` 不是引擎保留字

案例里经常这么写：

```javascript
var templateId = '';
templateld = i;
```

真正重要的是“前后一致”，不是这个拼写本身。

### 语法能力不要说得太死

真实案例里出现过：

- `Object.assign`
- `.find()`
- `.forEach()`
- 箭头函数
- `const`

所以：

- 默认仍可写保守语法
- 但不要再说“只能 ES5”

### `set()` 不是默认必需

模板说明里确实出现过：

- `getCheckBox(...)`
- `formatDateCN(...)`
- `headerUrl + t_xxx[0].val`

但真实脚本往往没有显式：

```javascript
set("getCheckBox", getCheckBox);
set("formatDateCN", formatDateCN);
```

规则：

- 先按已提炼案例默认行为处理
- 只有运行时缺符号，再补 `set()`

## 2. 返回根的三种主结构

### `formJs`

适合“模板逻辑重、脚本只补几个聚合字段”的情况。

典型信号：

- 多页报告
- 模板里大量内联 JS
- 脚本只补列表或标题字段

### 单个对象，内含多个数组

适合“一份总报告 + 多个数据表”。

典型信号：

- 模板里同一页有多个数据区
- 同一份头部服务多个 `<data>` 表
- 常见数组名：
  - `arr1`
  - `arr2`

### `outputData[]`

适合“每个数组元素对应一份报告实例”。

典型信号：

- 每个样品一页
- 每个地点一页
- 每个签名组合一页
- 一个模板页自动重复成很多续表页

不要默认把多页报表写成：

```javascript
{ page1: ..., page2: ... }
```

在已提炼案例里，主流是单对象或对象数组。

## 3. 占位符类型与选择顺序

真实案例里常见三类占位符：

- `=key`
- `${key}`
- 模板内联 JS / 表达式

另外 `=` 不只表示“纯字段”，也可能是真表达式：

```javascript
=t_dwName?t_num:''
=headerUrl + t_TestMan[0].val
```

### 最终输出 key 的确定顺序

先定“模板需要什么”，再定“值从哪里来”。

- 第一步：从模板图片逐个抄出占位符和内联表达式里真正被访问的 key。
- 第二步：JS 的最终输出 key 必须先满足这些模板 key。
- 第三步：再用结果图和 JSON 判断这些 key 应该给文本、对象还是数组。

这意味着：

- 如果模板写的是 `=t_rsj`，脚本就应该产出 `t_rsj`，不要自作主张写成 `t_rsJ`。
- 如果模板写的是 `=headerUrl + t_shuMan[0].val`，脚本应优先保留 `t_shuMan` 的原始数组结构，不要因为别的案例里常见 `t_TestMan` 就自动替代。
- 历史案例里的字段名只能帮助理解结构，不能直接拿来当当前模板的最终输出 key。
- JSON 用来确认“值从 `form` / `sample` / `gauging` 哪一层取”，不是用来发明模板里没有的新 key。
- 模板输出 key 和真实 source path 可能不同名，例如模板要 `t_two`，但当前实验室 JSON 里真正的值可能挂在 `gauging.two`。
- 这类 source path 往往是当前实验室自己配置的标签名，只能在当前任务里使用，不能总结成跨实验室通用规则。
- 当前任务标题、模板标题和结果图已经明确报表家族时，不能因为结构 JSON 里缺少同名模板，就自动改用“最像的邻近模板家族”。
- 结构 JSON 如果只提供了近邻模板，应只把它当字段形状和 source path 参考，不把它当当前任务身份依据。

### 精确过滤优先，禁止静默模糊匹配

如果当前材料已经出现或能够验证这些过滤条件：

- `sample.tempId`
- `gauging.gaugingTemplateName`
- `gauging.t_version`
- 其他能唯一缩小到目标报表家族的精确条件

则写脚本时优先使用精确过滤。

如果你怀疑“这里本来应该有精确过滤条件”，但当前材料还无法确认：

- 不要直接改成“名字里包含某几个字就算匹配”
- 不要静默用字段存在性来兜底匹配整类记录
- 要在交付说明里明确标成高风险，必要时追问用户

模糊匹配只能作为临时调试手段，不能默认当正式交付逻辑。

如果当前任务已经明确是 `137...` 这类报表：

- 那精确过滤首先应围绕 `137...` 这一家族去写
- 不能因为结构 JSON 里只有 `131...` 或别的近邻模板，就默认把正式过滤改成 `131...`

补充边界：

- 不要因为测试数据里恰好所有目标样品都属于同一个 `tempId`，就自动把这个 `tempId` 写进正式脚本。
- 只有当当前模板、结果图、占位符说明或最近邻家族骨架明确表明“这份报告只面向某一类样品模板”时，`sample.tempId` 才是正式过滤条件。
- 相比之下，`gauging.gaugingTemplateName` 更常直接代表报表家族本体；如果参考家族只用它过滤，就不要额外叠加一层 `tempId`。
- 要区分“引擎运行态输入 / 系统分配 ID”和“报表业务过滤字段”：前者如 `projectId` 不应硬编码，后者只有在有证据时才写死。
- 还要区分“结构键”和“展示键”：
  - `t_changSuo`、`sampleId`、`t_version`、`gaugingTemplateName` 这类键会参与分组、匹配、挂载明细
  - 这类键一旦当前家族已知，优先按家族精确写法取值，不要像 `t_tianqi`、`t_qiwen` 这类展示字段那样随意加别名回退
  - 否则看起来只是“更稳健的兼容”，实际会把不同地点/不同样品/不同检测家族错误合并到一份报告里

### 模板强依赖报告优先保留原始合并对象

如果当前任务更像这些情况：

- 单个 `reports` 对象承载整份报告
- 同一页有多个数据区
- 模板里可能还藏着截图中看不清的占位符或表达式
- 模板直接访问数组、对象或隐式继承的头字段

则默认优先：

1. 先把 `sample + form + gauging` 合并到一个原始报告对象里
2. 再补 `arr1`、`arr2`、`gaugingList` 之类显式明细数组
3. 最后只对确认安全的字段做预处理

只有在你能证明模板只依赖少量白名单字段时，才适合从一开始就手工构造纯白名单壳。

### `outputData[] + gaugingList` 家族的默认骨架

对这类特征明显的报告，优先考虑下面的真实骨架：

- 每个样品一页
- 页头字段是扁平文本
- 明细区是单个 `gaugingList`
- 模板不直接深挖复杂关联对象

默认优先：

1. `partData()` 定义页头白名单字段
2. `gaugingData()` 定义行白名单字段
3. `ObjAssign()` 负责按 key 从 `sample` / `gauging` / `form` 扁平取值
4. `getValue()` 统一处理数组、对象、签名 URL 和普通值
5. 第一轮先为每个样品创建输出壳
6. 第二轮再把对应 `gaugingList` 塞进去

补充规则：

- `partData()`、`gaugingData()` 只是定义“最终输出里有哪些 key”，不是在声明这些 key 只能从某一层数据源取。
- 行级 `gaugingList` 也不要默认写成“所有字段只从当前 gauging 取”；常见稳妥写法是先让 `form` / `sample` 给出默认值，再让当前行 `gauging` 覆盖。
- 换句话说，schema 白名单和 source precedence 是两个维度，不能因为用了白名单骨架，就把来源层级简化丢了。
- 这个骨架不只是几个 helper 名字相似而已，通常还隐含着：
  - 精确过滤锚点写在哪里
  - 先建页头输出壳还是先铺行数据
  - `gaugingList` 是单轮生成还是双轮追加
  - merge precedence 是 `form -> sample -> gauging` 还是别的顺序
  - 页序和行序是否保持原始顺序
- 因此一旦确认当前任务属于这个家族，默认策略不是“重新抽象出一套更通用的 buildReport/buildRows”，而是尽量小地偏离已证实骨架。

这不是唯一写法，但它是当前样本里最稳定、最可迁移的一类默认骨架。

### 不要过度工程化重写骨架

在这类 LabsCare 报表里，很多偏差不是“字段取错”，而是“控制流漂移”：

- 明明已经识别出最近邻家族，却又自己发明了一套新的函数分层
- 明明参考家族是两轮构建，却重写成单轮聚合
- 明明参考家族默认保留原始顺序，却顺手加了排序
- 明明参考家族是精确锚定当前模板名，却加了邻近模板兜底

这些改动表面上像“更稳健”，实际上会带来隐藏风险：

- 模板里未显式展示的字段继承链被改掉
- 页头和行数据的默认覆盖顺序被改掉
- 同家族报告之间的行为一致性被改掉
- 将来拿标准答案做对照时，难以判断是“业务差异”还是“作者自行重构”

因此默认策略应当是：

1. 先识别最近邻家族骨架
2. 再按当前模板/结果图做最小必要改动
3. 非证据驱动的“优化”先不要加

可以把这个策略理解成：

- 优先“derive by deletion / substitution”
- 而不是“synthesize from scratch”

### 不要发明默认排序

排序和去重不是“代码更整洁”的默认操作，而是报表业务行为。

规则：

- 只有结果图、任务说明或当前报表家族参考明确要求顺序时，才添加排序
- 如果材料没有给出顺序依据，默认保留原始数据顺序
- 尤其是 `outputData[]` 的页序，不要因为样品编号看起来可排序，就擅自重排

### 选择顺序

#### 第一选择：简单字段占位符

优先：

- `=key`
- `${key}`

适合：

- 文本
- 数值
- 已在脚本里预处理好的日期
- 已在脚本里预处理好的签名 URL

#### 第二选择：脚本先算好，再给简单占位符

优先把这些逻辑放脚本里：

- 因子名拼接
- 去重排序
- 固定行数补空
- 已处理好的图片 URL
- 已处理好的摘要标题

#### 第三选择：模板格子里写内联 JS

只在这些场景用：

- 模板必须访问原始数组/对象
- 单元格里只需要做轻量条件判断
- 现有模板版式已经大量依赖内联 JS

## 4. 表格循环与分页判断

要先分清两层循环：

### 报告级循环

例如：

- `outputData` 里每个元素一份报告
- 每个样品一份
- 每个地点一份
- 每个签名组合一份

### 行级循环

例如：

- `gaugingList`
- `arr1`
- `arr2`
- `newData`

### 如何反推

1. 先看模板图里哪一行像“单行原型”
2. 再看结果图里这块是否被展开成多行
3. 再判断脚本应返回哪一个数组供它循环

不要因为两个表格区看起来相似，就把一个表格区的字段回退到另一个表格区的字段。

例如：

- 模板上半区如果写的是 `t_bddwma`，下半区写的是 `t_dwName`
- 那脚本应分别产出 `t_bddwma` 和 `t_dwName`
- 不能为了“减少空值”就让 `t_bddwma <- t_dwName` 或 `t_dwName <- t_bddwma`

### 分页判断

已提炼案例证明：

- 一个模板页可以渲染出多页结果
- 续表未必是新模板，可能只是同页模板重复渲染
- 补空行规则会直接影响页数

所以页数判断必须参考结果图，而不是只看模板页数。

## 5. 模板内联 JS 的使用边界

模板内联 JS 不是默认方案，是最后兜底方案。

### 典型模板内联 JS

复选框：

```javascript
//javascript
var listText = ['石墨炉原子吸收法','火焰原子吸收法']
var list = []
var val = t_clff ? t_clff : ''
listText.forEach(function(e) {
    list.push(getCheckBox(val == e) + e)
})
list.join("&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;")
```

日期：

```javascript
${t_wordManDate ? formatDateCN(t_wordManDate) : ''}
```

签名图：

```javascript
=headerUrl + t_TestMan[0].val
```

### 什么时候必须保留原始结构

如果模板要这样写：

- `t_TestMan[0].val`
- `t_shuMan[0].val`
- `t_qx.caseList[0].t_JZtab[0]...`

那脚本就不能先把这些字段压成纯字符串。

### 给用户模板修改建议时怎么写

必须同时说明：

1. 哪一格需要改
2. 直接粘贴什么代码
3. 脚本侧哪些字段要保持原始结构

不要只说“这里建议用内联 JS”。

## 6. 交付前自检

至少逐项确认：

- 模板图里的每个关键占位符，都知道它是：
  - `=key`
  - `${key}`
  - `=表达式`
  - `//javascript`
- 结果图里的每个循环区，都能对上脚本里的某个数组字段
- 结果图里的每个签名位、图片位、日期位都能解释
- 直接给模板的字段类型是对的：
  - 字符串
  - 数组
  - 下拉对象
  - 关联对象
- 你能解释为什么返回根是：
  - `formJs`
  - 单对象
  - `outputData[]`

### 只有这些情况才追问用户

- 模板和结果图都看不出隐藏循环绑定名
- 关键字段来源无法判定
- 图片/签名 URL 方案无法从材料中判断

其他情况优先自行反推完成。

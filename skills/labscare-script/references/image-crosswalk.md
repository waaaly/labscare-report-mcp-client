# 模板图-脚本-结果图对照矩阵

这份表不是跨实验室字段字典，而是从 9 组真实报告里提炼出来的“图像关系速查表”。

## 目录

- 9 组报告的总表
- 每组案例的模板页、结果页和循环提醒
- 如何按任务类型选最近邻结构

使用方式：

1. 先拿当前任务的模板图和结果图，对照下面哪一组最像。
2. 再去读这份 skill 里的 `examples.md`、`patterns.md` 和本文件对应小节。
3. 如果当前任务还给了真实数据 JSON，再用 JSON 只做“字段类型确认”，不要把别组字段名硬搬过来。

## 样本级提醒

### `033水样采样记录表`

- 这组不是“每个样品一页”，而是“同签名组合的多个样品合成一份采样记录”。
- 模板里的多行表格更像是对 `ss` 的循环，而不是对 `gaugingTableList` 的直接循环。
- `ss[*].gaugingTableList` 仍要保留原始数组，因为模板内联 JS 会根据它生成“分析项目”和“备注”。
- 输出对象里同时保留 `p`、`s`、`ss`、`gs` 是有意设计，不是冗余字段。

### `092α、β表面污染测量结果处理表`

- 同一个模板页里有 2 个数据区，所以脚本要同时喂 `arr1` 和 `arr2`。
- 这组非常适合拿来判断“单对象 + 多个 `<data>` 表”的写法。
- 模板直接用 `=headerUrl + t_shuMan[0].val`、`=headerUrl + t_shenMan[0].val` 拼签名图。
- 如果把签名数组提前压扁成字符串，模板格子也必须跟着改；两边不能混写。
- 结果图能看到字段类型不匹配时的异常信号，说明对象/数组字段不能被当普通文本乱塞。

### `099中子辐射剂量率测量结果处理表`

- 这是“公共头 + 单个大表 + 自动续页”的典型。
- 当前样本把签名数组在脚本里提前转成图片 URL，所以模板里用简单字段 `=t_TestMan`、`=t_checkMan`。
- 时间字段在脚本里裁成 `HH:mm`，模板只负责展示。
- `=t_dwName?t_num:''` 说明 `=` 并不只绑定字段，也能直接写三元表达式。

### `137流出物样品γ谱分析原始记录表`

- 这组更像“每个样品单独出一页分析原始记录”，不是一个大总表。
- 两张模板图里，`page1` 更像说明/辅助图，`page1-1` 更像实际占位符模板；要结合看。
- `sampleId` 只是脚本里把 `gaugingList` 归到对应样品的匹配键，不一定是模板要直接用的占位符。
- 这个案例原始来源里目录名尾部带空格、脚本文件名大小写异常；这里已经完成了规则提炼，实际使用 skill 时不需要再处理这些路径细节。

### `X-γ辐射剂量率测量记录表`

- 这组要先按 `t_changSuo` 聚合，再往对应报告对象里塞 `gaugingList`。
- 结果图说明一个点位不是简单“1 项 = 1 行”，而更像“1 个点位对象 = 2 行视觉块”。
- 所以写新脚本时不要误把同一测点拆成互不相关的独立行。
- 模板里已有内联时间拼接 `${t_jcKStime}-${t_jcJStime}`，以及序号表达式 `=t_dwName?t_num:''`。

### `原子吸收光谱测定水样中元素分析原始记录表`

- 第 1 页不是普通头部，它吃的是从关联曲线对象 `t_glQX.fkCases[0].t_qx` 组装出来的 `newData`。
- 第 2 页才是样品结果表，对应 `gaugingList`，并且会反复续页。
- 这组说明“一个输出对象里可以同时挂 2 套不同用途的数组”。
- 模板页里有复选框内联 JS `getCheckBox(...)`，说明有些版式判断留在模板里比放脚本里更稳。
- 如果用户提供了真实数据 JSON，这组特别值得用来核对关联曲线字段是否真的是对象而不是字符串。

### `氡析出率测量结果处理表`

- 这是“同一页模板重复渲染成很多续表页”的最强示例之一。
- 样本里特意把 `t_htName`、`t_wtName` 强制改为 `form` 的值，说明公共头并不总能完全靠 `ObjAssign` 自动合并正确。
- `gaugingList` 还会被补空对象到固定倍数，所以结果页数不仅受真实记录数影响，也受补空行规则影响。
- 结果图被拆成 6 张 PNG，但它们实际上是同一份脚本执行出来的 1 份报告。

### `电感耦合等离子体质谱法分析原始记录表`

- 这组是“单对象 + 2 页模板 + 模板强依赖原始对象结构”的代表。
- `arr2` 才是真正的样品结果行；`arr1` 在当前样本里没有填充，但依旧保留为模板绑定位。
- 模板第 1 页直接内联访问 `t_qx.caseList[0]...`，所以 `t_qx` 必须保留原始对象，不能先被脚本压成纯文本。
- 同理，签名位用的是 `=headerUrl + t_TestMan[0].val` 这类模板表达式，就要求 `t_TestMan`、`t_TestManTwo`、`t_checkMan` 仍然是原始数组。
- 这组非常适合教用户“什么时候应该让模板读原始对象，什么时候应该让脚本先算出 `t_dw` / `t_nbdw` 这类简单字段”。

### `监测报告（电磁）`

- 这是当前样本里模板依赖最高的一组，脚本只负责把 `formJs` 补到模板够用。
- 第 2 页几乎是静态版式，第 3 页是否出现 1 个还是 2 个监测项目表，要看 `monitoringItemSubTitle`。
- 第 4 页的图片、签名、中文日期全部依赖模板内联 JS。
- 这组里 `t_xmType`、`t_tu`、`t_wordMan`、`t_wordCMan`、`t_wordMMan` 以及日期字段都不能随便压扁，否则模板格子会失效。

## 什么时候优先参考哪组

- 需要“单对象 + 多个明细表”：先看 `092`、再看 `电感耦合`。
- 需要“公共头 + 一个大表自动续页”：先看 `099`、`氡析出率`、`原子吸收`。
- 需要“每个样品各出一份”：先看 `137`。
- 需要“先分组再出报告”：按签名分组看 `033`，按地点分组看 `X-γ`。
- 需要“模板自己做很多标题/图片/日期判断”：直接看 `监测报告（电磁）`。

## 用这份矩阵时要记住

- 这份表能迁移的是“结构判断方法”，不是字段名字典。
- 页数关系比字段名更值得先看，因为它直接决定你该返回单对象、对象数组，还是 `formJs`。
- 一旦模板里出现 `=xxx[0].val`、`Object.prototype.toString.call(...)`、`//javascript`，就要先停一下，检查脚本有没有把原始对象结构压坏。
